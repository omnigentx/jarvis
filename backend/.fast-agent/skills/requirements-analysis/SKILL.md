# Requirements Analysis (BA)

> **⚠️ RESERVED:** This skill is prepared for the future BA role. Not currently in use.

## Purpose

Guide the Business Analyst role in:
- Gathering and clarifying requirements from stakeholders
- Translating business needs into technical specifications
- Writing acceptance criteria
- Managing scope and priorities

## Requirements Process

### 1. Stakeholder Analysis
- Identify all stakeholders and their interests
- Understand business goals and constraints
- Map relationships between stakeholders

### 2. Requirements Gathering
- Ask open-ended questions to understand the "why"
- Document user stories: "As a [role], I want [action] so that [benefit]"
- Identify assumptions and constraints
- Prioritize: Must-have vs. Nice-to-have (MoSCoW method)

### 3. Acceptance Criteria
For each requirement, define:
- **Given** [precondition]
- **When** [action]
- **Then** [expected result]

### 4. Gap Analysis
- Compare current state vs. desired state
- Identify missing capabilities
- Estimate effort and impact

### 5. Documentation
- Requirements specification document
- Use case diagrams
- Process flow diagrams
- Traceability matrix

## Quality Checks

- Is each requirement **testable**?
- Is each requirement **unambiguous**?
- Are requirements **complete** (no gaps)?
- Are there any **contradictions**?
- Is the **scope** clearly bounded?

## In Meeting Context

As BA in a meeting:
- Present requirements clearly with acceptance criteria
- Validate that DEV's implementation matches requirements
- Clarify ambiguities when team has questions
- Flag scope creep early
